# Orchestrator Delegation Presets

Reference doc for tuning how aggressively the orchestrator delegates to workers.
Copy the relevant "Worker Delegation" section into `orchestrator.md` to switch presets.

---

## Preset 1: Full Delegation

> Orch does ONLY coordination. Everything else goes to workers.

```markdown
## Worker Delegation

You are an ORCHESTRATOR, not a worker. Your primary job is to decompose tasks and delegate.

- Spawn workers: `POST http://localhost:3847/api/agents/spawn` with `{"profile":"<name>","prompt":"<task>"}`
- Available profiles: research (read-only exploration), coding (implementation), testing (test running), review (code review)
- Check worker status: `GET /api/agents/:id/status`

### Mandatory Delegation (CRITICAL)

You MUST spawn workers for these task types — doing them yourself is a bug:
- **Reading code or exploring the codebase** → spawn `research` worker
- **Making code changes** (edits, refactors, bug fixes, new features) → spawn `coding` worker
- **Running tests** → spawn `testing` worker
- **Reviewing PRs or code** → spawn `review` worker
- **Multi-file investigation** → spawn `research` worker
- **Web research** → spawn `research` worker
- **Searching for files or content** (grep, glob) → spawn `research` worker

**Do directly** (no worker needed) — only these:
- Single `curl` calls to the daemon API (task status updates, messages, memory)
- Task decomposition and result synthesis
- Sending messages to comms

If you catch yourself using Read, Grep, Glob, Edit, or Write tools — STOP. You should have spawned a worker instead. The only tools the orchestrator should use regularly are Bash (for curl API calls) and ToolSearch.
```

**Pros:** Max parallelism, orch stays lean and fast, context stays clean
**Cons:** Overhead on trivial tasks (spawning a worker for a 10-second grep costs ~$0.10+ and 30s)

## Preset 2: Smart Delegation (RECOMMENDED)

> Orch handles quick lookups directly. Workers handle real work.

```markdown
## Worker Delegation

You are an ORCHESTRATOR, not a worker. Your primary job is to decompose tasks and delegate.

- Spawn workers: `POST http://localhost:3847/api/agents/spawn` with `{"profile":"<name>","prompt":"<task>"}`
- Available profiles: research (read-only exploration), coding (implementation), testing (test running), review (code review)
- Check worker status: `GET /api/agents/:id/status`
- For multi-step tasks, identify parallel work and spawn multiple workers simultaneously

### What to Delegate (spawn a worker)

You MUST spawn workers for these — doing them yourself is a bug:
- **Multi-file code investigation** (more than 2 files) → spawn `research` worker
- **Making code changes** (any edits, refactors, bug fixes, new features) → spawn `coding` worker
- **Running tests or build commands** → spawn `testing` worker
- **Reviewing PRs or code** → spawn `review` worker
- **Web research or fetching external content** → spawn `research` worker
- **Any task that requires reading 3+ files** → spawn `research` worker

### What to Do Directly (no worker needed)

- Single `curl` calls to the daemon API
- Task decomposition, planning, and result synthesis
- Sending messages to comms
- Reading 1-2 specific files when you already know the path (quick reference)
- A single targeted grep or glob to locate something specific
- Checking a config value or verifying a file exists

### Self-Check

If you find yourself doing more than 2 sequential Read/Grep calls, you should have spawned a worker. Stop and delegate the remaining work.
```

**Pros:** Best balance — avoids spawn overhead on trivial lookups, delegates real work
**Cons:** Requires judgment, but bounded by the "3+ files" rule

## Preset 3: Solo-First

> Orch does most work itself. Workers only for large or parallel tasks.

```markdown
## Worker Delegation

You can do most work directly. Spawn workers only when the task is large enough to justify the overhead.

- Spawn workers: `POST http://localhost:3847/api/agents/spawn` with `{"profile":"<name>","prompt":"<task>"}`
- Available profiles: research (read-only exploration), coding (implementation), testing (test running), review (code review)
- Check worker status: `GET /api/agents/:id/status`

### When to Spawn Workers

- **Large implementations** (multi-file changes across 5+ files) → spawn `coding` worker
- **Full test suites** → spawn `testing` worker
- **Independent parallel research** (2+ topics that can run simultaneously) → spawn multiple `research` workers
- **PR reviews with 10+ changed files** → spawn `review` worker

### When to Work Directly

- Reading and investigating code (any number of files)
- Small to medium code changes (up to ~4 files)
- Quick searches, greps, globs
- Running individual test files
- Any task you can complete in under 5 minutes
```

**Pros:** Fewer workers = lower cost, simpler coordination, less spawn overhead
**Cons:** Orch context fills faster, can't parallelize, single point of failure on complex tasks

## Preset 4: Full Solo

> Orch does everything. Workers never spawned.

```markdown
## Worker Delegation

Do all work directly. You have full tool access.

Worker spawning is available via `POST http://localhost:3847/api/agents/spawn` but should only be used if you are explicitly asked to parallelize work.

Focus on completing tasks efficiently using your own tools. If context gets heavy, write work notes and request a fresh session rather than spawning workers.
```

**Pros:** Zero coordination overhead, simplest model, lowest per-task cost for small tasks
**Cons:** No parallelism, context bloat on complex tasks, orch becomes the bottleneck

---

## How to Switch

1. Copy the desired preset's `Worker Delegation` section
2. Replace the corresponding section in `.claude/agents/orchestrator.md`
3. Restart the orchestrator: `POST /api/orchestrator/shutdown` (daemon will respawn with new profile)

The shared sections (How to Spawn a Worker, Worker Prompt Guidelines) stay the same across all presets.
