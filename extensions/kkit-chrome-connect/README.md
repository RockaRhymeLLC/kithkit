# KithKit Chrome Connect

A Manifest V3 Chrome extension that bridges Chrome's debugger API to a KithKit daemon over an encrypted WebSocket. Minimum Chrome version: **Chrome 114** (required for the MV3 offscreen document API).

## Installation

1. Open `chrome://extensions`
2. Enable **Developer mode** (toggle, top right)
3. Click **Load unpacked**
4. Select the `kkit-chrome-connect/` directory

The extension icon appears in the toolbar. A badge shows connection state: no badge = disconnected, `ON` = connected, `…` = connecting, `!` = error.

## PSK Setup

The Pre-Shared Key (PSK) is a 256-bit secret that authenticates and strengthens the encrypted session. Both the daemon and the extension must use the same PSK.

### Daemon side

The PSK is stored in macOS Keychain and referenced by the daemon. Set it via the daemon API:

```bash
# Store PSK in the daemon (replace with your 64-char hex string)
curl -s -X POST http://localhost:3847/api/cowork/psk \
  -H "Content-Type: application/json" \
  -d '{"psk": "your-256-bit-hex-psk-here"}'
```

To generate a fresh PSK:

```bash
openssl rand -hex 32
```

The cowork bridge must be enabled in `kithkit.config.yaml`:

```yaml
extensions:
  cowork-bridge:
    enabled: true
```

### Extension side

1. Click the extension icon in the toolbar
2. Expand the **PSK** section (click the toggle)
3. Paste your 64-character hex PSK into the PSK field
4. The PSK is saved in `chrome.storage.local` — it persists across browser restarts

## Auth Token

The auth token is a 256-bit hex value generated by the daemon. It is checked before the key exchange and provides a first layer of access control.

Generate or rotate a token:

```bash
curl -s -X POST http://localhost:3847/api/cowork/token/rotate
```

Copy the returned token and paste it into the **Token** field in the extension popup. The token is stored in `chrome.storage.session` — it is cleared when the browser session ends.

## Connecting

1. Click the extension icon
2. Enter the daemon host in the **Host** field:
   - Local: `localhost:3847`
   - Cloudflare tunnel: `cowork.bmobot.ai` (uses `wss://` automatically)
3. Enter the **Token** (from `POST /api/cowork/token/rotate`)
4. Expand **PSK** and enter the hex PSK if encryption is required
5. Click **Connect**

On success the badge turns `ON` and the popup shows the connected host, the attached tab title and URL, and the daemon's key fingerprint (first 16 hex chars of SHA-256 of its public key).

## Architecture

```
KithKit Daemon
     |
     | WebSocket  ws://<host>:<port>/cowork
     |            wss://cowork.bmobot.ai/cowork
     |
Extension Background Service Worker  (background.js)
     |
     | chrome.debugger API  (CDP 1.3)
     |
Active Chrome Tab
```

### Message flow

```
1. WebSocket connect
2. Auth         Client → Daemon  { type: "auth", token }
                Daemon → Client  { type: "auth-ok" }

3. Key exchange Client → Daemon  { type: "key-exchange", publicKey: <base64 X25519> }
                Daemon → Client  { type: "key-exchange", publicKey: <base64 X25519> }

4. Session key  Both sides run:
                  salt = HMAC-SHA256(PSK, "kkit-cowork-e2e-v1")
                  info = sorted([extPubKey, daemonPubKey]).join(":")
                  key  = HKDF-SHA256(ECDH shared bits, salt, info, 32 bytes)
                → AES-256-GCM session key (non-extractable)

5. Encrypted hello
                Client → Daemon  encrypted{ type: "hello", userAgent }  seq=1
                Daemon → Client  encrypted{ type: "hello-ack" }         seq=1

6. Encrypted session (all subsequent messages in AES-256-GCM envelopes)
                Daemon → Client  encrypted{ type: "cdp", id, method, params }
                Client → Daemon  encrypted{ type: "cdp-result", id, result }
                Client → Daemon  encrypted{ type: "cdp-error", id, error }
                Client → Daemon  encrypted{ type: "cdp-event", method, params }
                Daemon → Client  encrypted{ type: "list-tabs", id }
                Client → Daemon  encrypted{ type: "tab-list", id, tabs[] }
                Daemon → Client  encrypted{ type: "switch-tab", id, tabId }
                Client → Daemon  encrypted{ type: "tab-switched", id, tabId }
                Client → Daemon  encrypted{ type: "tab-changed", tabId, title, url }
                Either           encrypted{ type: "ping" } / { type: "pong" }
```

Each encrypted envelope is `{ type: "encrypted", seq, payload, nonce }` where `payload` is base64(ciphertext + GCM auth tag) and `nonce` is a random 12-byte base64 IV. The AAD is `"cowork:<seq>"`, binding each message to its sequence number to prevent replay.

If no PSK is configured the session falls back to plaintext after auth. Token-only connections are permitted but unencrypted.

### Keepalive

MV3 service workers are suspended by Chrome when idle. The extension uses two layers to stay alive during a connected session:

- **Offscreen document** (`keepalive.html`) — pings the service worker every 25 seconds via `chrome.runtime.sendMessage`
- **chrome.alarms** — fires every 30 seconds and sends an encrypted `ping` to the daemon

Both are created on connect and destroyed on disconnect.

### Debugger attachment

When connected, the extension attaches Chrome's debugger to the active tab using `chrome.debugger.attach`. It follows the user as they switch tabs — detaching from the previous tab and attaching to the new one automatically.

**Note:** When `chrome.debugger` attaches, Chrome displays a yellow infobar at the top of every window: **"DevTools is debugging this browser."** This is a Chrome security requirement and cannot be suppressed. It disappears when the extension disconnects.

## Security

- **X25519 ECDH** — ephemeral keypair generated per session; forward secrecy
- **PSK-HKDF** — the PSK is mixed into the HKDF salt via `HMAC-SHA256(PSK, "kkit-cowork-e2e-v1")`, so a stolen ECDH key alone cannot decrypt traffic
- **AES-256-GCM** — authenticated encryption; any tampering causes decryption to throw and the connection to close
- **Sequence numbers** — monotonically enforced in both directions; any gap or repeat closes the connection (`4003 Sequence violation`)
- **TOFU fingerprint** — the popup displays the first 16 hex chars of `SHA-256(daemon public key)` after key exchange; verify this matches on both sides when setting up for the first time
- **Auth token** — 256-bit random token required before key exchange; limits connections to clients that know the token
- **Localhost only** — the daemon binds to `127.0.0.1` by default; the cowork endpoint is not exposed remotely unless you configure a tunnel

## Permissions

| Permission   | Why                                                            |
|--------------|----------------------------------------------------------------|
| `debugger`   | Attach to tabs and relay CDP commands                          |
| `activeTab`  | Access the currently active tab on Connect                     |
| `storage`    | Persist host and PSK (local); token and connection state (session) |
| `tabs`       | Listen for tab switches/removals, query tab list               |
| `alarms`     | Keepalive heartbeat to prevent service worker suspension       |
| `offscreen`  | Create the keepalive offscreen document (requires Chrome 114+) |

## Troubleshooting

**Connection times out immediately**
- Confirm the daemon is running: `curl http://localhost:3847/health`
- Confirm `cowork-bridge.enabled: true` in `kithkit.config.yaml`
- If using a Cloudflare tunnel, check the tunnel is active

**"Auth failed: expected auth-ok"**
- The token in the popup does not match the daemon's current token
- Rotate the token: `curl -X POST http://localhost:3847/api/cowork/token/rotate`
- Paste the new token into the popup before connecting

**"Hello handshake failed" or "Decryption failed"**
- The PSK in the popup does not match the PSK stored in the daemon
- Verify both sides use the same 64-character hex string
- PSK must be exactly 64 hex characters (256 bits)

**"Sequence violation"**
- The session was desynchronized (network hiccup, partial message, or replay attempt)
- Click Disconnect, then Connect again to start a fresh session

**Yellow "DevTools is debugging this browser" infobar**
- This is expected behavior from Chrome whenever `chrome.debugger` is attached
- It cannot be suppressed — it is a Chrome security notice, not an error

**Extension badge stuck on `!`**
- An error occurred during connection; the badge auto-clears after 3 seconds
- Open the popup to see the error message

**Service worker suspended while connected**
- The dual keepalive (offscreen document + alarms) should prevent this
- If it happens, click Disconnect and reconnect; the keepalive layers restart on connect
- Requires Chrome 114+ for the offscreen document API

**Debugger detached unexpectedly**
- Another tool (DevTools, another extension) may have attached its own debugger to the tab
- Chrome only allows one debugger per tab; open DevTools will conflict
- Close DevTools and reconnect
